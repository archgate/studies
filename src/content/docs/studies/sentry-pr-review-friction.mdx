## Abstract

> This study evaluates where review back-and-forth is concentrated in `getsentry/sentry` and identifies which Architecture Decision Records (ADRs) plus enforceable rules could reduce repeated discussion cycles. The analysis covers merged, abandoned, and stale pull requests to capture the full spectrum of governance friction.

## 1. Scope & methodology

| Parameter | Value |
|-----------|-------|
| Window | Last 90 days |
| Sample (merged) | 500 PRs |
| Sample (closed-unmerged) | 500 PRs |
| Sample (open) | 286 PRs |
| Metrics | Review events, merge latency, changed file count, churn |
| Deep dive | Highest-friction PRs by review-event volume and merge latency |

## 2. Baseline findings

<Aside type="note" title="Merge-bias caveat">
The baseline below covers **merged PRs only**. It captures friction in successfully landed work but misses high-discussion PRs that were closed without merge and open PRs with substantial discussion that later went stale. Those cohorts are analyzed in Section 3.
</Aside>

| Metric | Value |
|--------|-------|
| Median time-to-merge | `4.12 h` |
| P75 time-to-merge | `21.84 h` |
| P90 time-to-merge | `69.54 h` |
| Median review events / PR | `2` |
| Mean review events / PR | `3.19` |
| Formal `CHANGES_REQUESTED` | `1%` |
| Small PR median TTM | `2.07 h` |
| Large PR median TTM | `21.44 h` |

<div class="key-finding">
<strong>Key finding:</strong> Large PRs experience <strong>10×</strong> the median merge latency of small PRs (21.44 h vs 2.07 h), confirming PR size as a primary friction driver.
</div>

## 3. Missing-signal extension: abandoned and stale PRs

Measuring merged PRs alone understates governance friction. To surface hidden signals we expanded the analysis to two additional cohorts.

### 3.1 Closed-unmerged PRs

| Metric | Value |
|--------|-------|
| Queried | 500 closed-unmerged PRs |
| Median discussion volume | `2` |
| High-discussion abandoned share (≥10 items) | `2.4%` |

Notable examples:

| PR | Discussion items | Outcome |
|----|----------------:|---------|
| [#109150](https://github.com/getsentry/sentry/pull/109150) | 45 | Closed unmerged |
| [#108406](https://github.com/getsentry/sentry/pull/108406) | 30 | Closed unmerged |
| [#109526](https://github.com/getsentry/sentry/pull/109526) | 24 | Closed unmerged |
| [#111112](https://github.com/getsentry/sentry/pull/111112) | 22 | Closed unmerged |

### 3.2 Open stale PRs

| Metric | Value |
|--------|-------|
| Open PRs sampled | `286` |
| Stale ≥14 days | `36` |
| Stale ≥30 days | `9` |

Notable examples:

| PR | Discussion items | Stale (days) |
|----|----------------:|-------------:|
| [#108533](https://github.com/getsentry/sentry/pull/108533) | 42 | 20 |
| [#109781](https://github.com/getsentry/sentry/pull/109781) | 6 | 19 |

<div class="key-finding">
<strong>Key finding:</strong> Abandoned high-traction PRs and stale discussed PRs should be tracked as <strong>first-class outcomes</strong> in governance friction metrics — measuring merged PRs alone can mask unresolved decision ambiguity.
</div>

## 5. Recurring discussion themes

Across the highest-friction PRs, five discussion patterns recurred consistently:

1. **API contract and compatibility semantics** — debates about backward compatibility, endpoint behavior, and contract boundaries.
2. **Nullability / type safety and error paths** — repeated discussions on optional types, null checks, and error propagation.
3. **UX flow behavior invariants** — disagreements on user-facing behavioral expectations and edge-case handling.
4. **Testing evidence expectations** — requests for additional test coverage, missing test cases, or inadequate assertions.
5. **Permission and reliability guardrails** — concerns about access control, rate limiting, and failure-mode handling.

## 6. Proposed ADR pack

Based on the friction patterns identified, we propose five Architecture Decision Records:

| # | ADR | Addresses |
|---|-----|-----------|
| 1 | **PR Slice Boundaries and Risk Budget** | PR size → merge latency correlation |
| 2 | **API Contract Evolution Protocol** | API compatibility discussion cycles |
| 3 | **Test Evidence Matrix by Change Type** | Testing expectation mismatches |
| 4 | **UI/Flow Behavioral Invariants** | UX behavior disagreements |
| 5 | **Permission and Reliability Guardrails** | Access control and failure-mode debates |

<Aside type="tip" title="Adoption">
Each ADR is designed to be enforceable via automated checks. Teams can adopt the full pack or select individual records based on their highest-friction areas.
</Aside>
