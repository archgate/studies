**Repository:** getsentry/sentry · **Window:** 90 days · **Sample:** 500 merged PRs · **Status:** Published

> **Abstract** — This study evaluates where review back-and-forth is concentrated in `getsentry/sentry` and identifies which Architecture Decision Records (ADRs) plus enforceable rules could reduce repeated discussion cycles. We analyze 500 merged PRs over a 90-day window, extend the analysis to abandoned and stale PRs, and propose a five-ADR standardization pack targeting the highest-friction themes.

## Scope

| Parameter | Value |
|---|---|
| Time window | Last 90 days |
| Sample | 500 merged PRs |
| Metrics | Review events, merge latency, changed file count, churn |
| Deep dive | Highest-friction PRs by review-event volume and merge latency |

### Coverage Caveat

> The baseline above is **merge-biased**. It captures friction in merged PRs but misses: (1) high-discussion PRs that were closed without merge, and (2) open PRs with substantial discussion that later went stale. Those two cohorts often contain failed-consensus signals and unresolved decision ambiguity.

## Missing-Signal Extension: Abandoned and Stale PRs

## Recurring Discussion Themes

Analysis of the high-friction PRs reveals five recurring categories of review friction:

1. **API contract and compatibility semantics** — disagreements about backward-compatible changes, versioning, and contract boundaries
2. **Nullability/type safety and error paths** — review cycles around null handling, type narrowing, and error propagation
3. **UX flow behavior invariants** — debates about user-facing behavioral expectations and edge cases
4. **Testing evidence expectations** — friction over what constitutes sufficient test coverage for a given change type
5. **Permission and reliability guardrails** — discussion about authorization checks, rate limiting, and failure-mode handling

## Proposed ADR Pack

Based on the recurring themes, we propose five ADRs for `getsentry/sentry`:

| # | ADR Title | Target Theme |
|---|---|---|
| 1 | PR Slice Boundaries and Risk Budget | PR size → review effort correlation |
| 2 | API Contract Evolution Protocol | API compatibility semantics |
| 3 | Test Evidence Matrix by Change Type | Testing evidence expectations |
| 4 | UI/Flow Behavioral Invariants | UX flow behavior contracts |
| 5 | Permission and Reliability Guardrails | Auth & reliability conventions |
